High severity

S1

Committed Firebase credentials belong to an unrelated project

High Partially resolved

src/environments/environment.ts:10-17 · environment.prod.ts · environment.staging.ts

What
All three environments — including production — ship the Firebase config of redacted-firebase-project (authDomain: redacted-firebase-project.firebaseapp.com), a different product entirely, API key included.
Why
Push registration for com.gp.gp-group cannot work against that project, so FCM is broken at release. And a live key + project identifiers of another application sit in this repo's history — quota/abuse surface on the wrong project, and a guaranteed flag in any review.
Fix
Create a dedicated Firebase project for G.P. News, generate per-platform configs when android//ios/ are added, replace all three env blocks, and rotate/restrict the exposed key on the old project (API-key restrictions by app/package).

Status (2026-07-06): the firebase block was deleted from all three environment files, and the leaked API key / project id / sender id were purged from every commit in git history via git filter-repo — verified with zero remaining matches. A dedicated Firebase project has since been created and capacitor.config.ts's appId now matches its registered app id (com.gp.gp-group — bundle id was updated repo-wide, see CLAUDE.md). Still open: the exposed key on redacted-firebase-project has not been confirmed rotated/restricted in that project's Firebase/GCP console (external action, outside this repo), and no google-services.json / GoogleService-Info.plist or web config for the new project has been added to the repo yet — push notifications remain unconfigured until those land.

Medium severity

S2

No Content-Security-Policy on the WebView

Medium

src/index.html — no CSP meta anywhere in src/

What
The app renders API-driven HTML inside a Capacitor WebView with no CSP; the page also currently loads remote CSS from fonts.googleapis.com, which a strict policy would surface.
Why
Angular's sanitizer is the only layer between API content and the DOM. CSP is the standard defense-in-depth for Capacitor apps: it blocks inline script execution, unexpected remote loads, and exfiltration channels if any injection slips through.
Fix
After self-hosting fonts (perf audit H2), add default-src 'self' capacitor:; img-src 'self' data: https:; style-src 'self' 'unsafe-inline'; connect-src 'self' https://api.gpgroup.bg and tighten from there.
S3

Article HTML via [innerHTML] — sanitizer is the only guard

Medium

src/app/pages/article/article-detail/article-detail.page.ts:93

What
[innerHTML]="a.content" renders server-supplied HTML. Verified: zero bypassSecurityTrust/DomSanitizer usage in the codebase, so Angular's sanitizer applies — scripts, event handlers and javascript: URLs are stripped.
Why
The sanitizer still permits arbitrary <a href> (phishing links styled as app content) and <img src> to any host (tracking / remote content). A compromised CMS becomes a direct content-injection channel.
Fix
Keep the sanitizer (never bypass it), add the S2 CSP to control remote loads, and prefer server-side allowlist sanitization (p, h2, h3, a, img, ul, ol, li, b, i, blockquote) before content is stored.
S4

Push-notification tap navigates to an unvalidated URL

Medium

src/app/core/services/push-notification.service.ts:102-109

What
handleNotificationTap calls router.navigateByUrl(data['url']) with whatever string arrives in the push payload.
Why
Anyone able to send pushes (compromised backend, leaked FCM key — see S1) can deep-link users to any in-app route as a social-engineering primitive. navigateByUrl stays in-app so impact is bounded, but the payload is untrusted input.
Fix
Validate against a route allowlist, or accept only typed fields (articleId, pollId) and build routes locally — the articleId branch already does this correctly.
S5

Session restored without server validation; PII in plaintext storage

Medium

src/app/store/auth/auth.store.ts:169-205 · :92-101 · storage.service.ts

What
bootstrap() marks the user logged in purely from local metadata + token presence — AuthService.me() exists but is never called, so a server-revoked session looks valid until the first 401. The gpnews.authData blob (user object + expiry) lives in plain Ionic Storage, unlike the token, which correctly sits in the keychain.
Why
On a shared or compromised device the user's identity data is readable without the keychain, and server-side revocation (stolen device, forced logout) does not take effect at app start.
Fix
Fire-and-forget me() after optimistic restore and force logout on 401; move authData into secure storage, or store only the user id and refetch the profile.

Low severity

L1

Weak client-side password rules

Low

login.page.ts:115 minLength(4) · signup.page.ts:140 minLength(6)

What
No complexity/entropy checks; login accepts 4 characters, signup 6. Client validators are UX, not security — but they signal the server policy (mock backend today).
Fix
Real backend must enforce length ≥ 8, rate-limit /auth/login, and lock out brute force. Align the login validator with the real minimum so errors are accurate.
L2

External links in article HTML fall through to WebView defaults

Low

article-detail.page.ts:609-616

What
onEditorClick intercepts only internal (/…) hrefs and returns for everything else, leaving external https:// anchors to platform-dependent Capacitor defaults.
Fix
Handle them explicitly: open via @capacitor/browser (or window.open with noopener) so remote pages can never replace the app UI inside the WebView.
L3

returnUrl is produced but never consumed

Low

auth.guard.ts:16-18 stashes it · auth.store.ts:113 always navigates to /tabs/home

What
Today a UX gap; if wired up naively later it becomes an open-redirect-style primitive.
Fix
When implementing, accept only app-internal paths — reject absolute URLs and //-prefixed values.
L4

Native-layer hardening pending — platforms not yet added

Low

no android/ or ios/ directories · capacitor.config.ts clean (no server.url, no allowNavigation, no cleartext)

What
Checklist for when platforms are generated: cleartext traffic off in release (dev API is http://localhost:3000), android:allowBackup=false or backup rules, certificate-pinning decision for api.gpgroup.bg, ProGuard/R8 on, WebView debugging off in release.

Done well

  • ✓Token storage. Access token in iOS Keychain / Android EncryptedSharedPreferences; web dev fallback is in-memory only — explicitly never localStorage (secure-storage.service.ts:8-16).
  • ✓Scoped Bearer. Interceptor attaches the token only to apiBaseUrl requests — never third-party hosts (auth.interceptor.ts:26-28).
  • ✓Refresh flow. Single-flight serialized 401 refresh queue; failure forces logout and clears persisted state; web refresh uses an httpOnly cookie with withCredentials.
  • ✓No sanitizer bypasses. Zero bypassSecurityTrust usage; comments and names rendered via interpolation, not HTML.
  • ✓Forms. Correct autocomplete hints (email, current-password, new-password); typed inputs; exhaustMap blocks double-submit of login/register.
  • ✓Route protection. authGuard on add-news, guestGuard on auth screens — both pure UrlTree redirects.
  • ✓Session expiry. Auto-logout timer derived from token expiry, cleared correctly, re-checked at bootstrap.
  • ✓Logging hygiene. Only two console calls in the app (push diagnostics) — no tokens, credentials, or PII logged.
  • ✓No dangerous primitives. No eval, no document.cookie, no dynamic script injection.
  • ✓Capacitor config. Minimal and safe — no remote server.url, no allowNavigation wildcards.
